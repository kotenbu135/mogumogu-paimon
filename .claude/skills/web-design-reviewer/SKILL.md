# Web デザインレビュー

Web サイトの視覚的デザイン検査と問題修正を体系的に実施します。

## 対応プラットフォーム

- 静的サイト
- SPA（React / Vue / Angular / Svelte）
- フルスタックフレームワーク（Next.js / Nuxt）
- CMS プラットフォーム

## コア要件

- **ターゲットサイトの稼働**: ローカル/リモートサーバーで実行中
- **ブラウザ自動化**: スクリーンショット・DOM アクセス
- **ソースコードアクセス**: デザイン問題修正時に必要

## ワークフロー（4 フェーズ）

### 1. 情報収集

- URL を確認
- フレームワーク・スタイリング方式を検出（Tailwind CSS など）
- プロジェクト構造を理解

### 2. 視覚検査

複数のビューポイント（375px〜1920px）でテスト：

- **レイアウト**: オーバーフロー、重なり、アライメント
- **レスポンシブ**: モバイル対応、ブレークポイント
- **アクセシビリティ**: コントラスト、フォーカス状態、Alt テキスト
- **一貫性**: フォント、色、余白

### 3. 問題の分類と優先度付け

| カテゴリ | 内容 | 優先度 |
|---------|------|--------|
| レイアウト | オーバーフロー、重なり、アライメント | 高 |
| レスポンシブ | モバイル非対応、ブレークポイント問題 | 高 |
| アクセシビリティ | コントラスト、フォーカス、Alt テキスト | 中〜高 |
| 視覚的一貫性 | フォント、色、余白 | 中〜低 |

### 4. 問題修正と再検証

- 優先度に基づいて対処
- 最小限で的確なコード修正
- 修正後に回帰テスト実施

## 推奨実装

**Playwright MCP** を推奨：
- ナビゲーション
- DOM スナップショット
- スクリーンショット
- インタラクション

## ベストプラクティス

- 段階的に修正（一度に多数の変更はしない）
- 既存コードパターンを尊重
- 修正内容をドキュメント化
- ユーザー確認なしで大規模リファクタリングはしない

## Tailwind CSS を使用する場合

このリポジトリで Tailwind CSS 4 を使用している場合：

- 既存の color scheme を確認
- `tailwind.config.ts` の設定を尊重
- カスタムカラー・スペーシング定義を活用
- Responsive prefixes（sm:、md: など）を活用
