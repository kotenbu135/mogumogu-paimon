# パフォーマンス監査スキル

静的エクスポート（`out/`）のバンドルサイズを分析し、最適化ポイントを提示します。

## 手順

### 1. ビルド実行

```bash
cd webapp && npm run build
```

ビルドが成功したことを確認してから次のステップに進む。
ビルドエラーがある場合は修正してから監査を続行すること。

### 2. バンドルサイズ集計（サブエージェント）

Agent ツールを `context: fork` で起動し、以下の分析を委譲する:

```
webapp/out/ ディレクトリ内の全 JS・CSS ファイルのサイズを集計してください。

手順:
1. out/_next/static/ 配下の全 .js・.css ファイルを列挙する
2. 各ファイルのサイズ（バイト・KB）を計測する
3. サイズ降順でソートしてリスト化する
4. 合計サイズを算出する
5. 50KB を超えるファイルを「要調査バンドル」として別途リストアップする

出力形式:
- ファイルパス | サイズ(KB) | 種別(JS/CSS)
- 合計サイズ
- 50KB 超ファイル一覧
```

### 3. 大きなバンドルの原因特定（サブエージェント）

ステップ 2 で特定した 50KB 超ファイルを対象に、Agent ツールで以下を実行する（`context: fork`）:

```
以下の大きなバンドルファイルについて、原因コンポーネントを特定してください。

要調査バンドル: {{ステップ2の50KB超ファイル一覧}}

手順:
1. webapp/src/app/ および webapp/src/components/ 配下の全 .tsx/.ts を読み込む
2. 以下の観点で大きなバンドルの原因を調査する:
   a. クライアントサイドに含まれる大きなライブラリ（lodash / date-fns / chart.js など）
   b. "use client" が付いているコンポーネントの一覧と、その依存関係
   c. Server Component で完結できるのに "use client" になっているコンポーネント
   d. 動的インポートなしで静的に import しているコンポーネント

原因特定結果を以下の形式で出力:
- コンポーネントパス | 推定影響サイズ | 原因の種別
```

### 4. 最適化提案の出力

ステップ 2・3 の分析結果をもとに、以下のカテゴリ別に最適化提案を出力する:

#### 4-1. 動的インポート（next/dynamic）候補

Agent ツールで以下を実行する（`context: fork`）:

```
webapp/src/ 配下のコンポーネントから、next/dynamic での動的インポートに適した候補を特定してください。

判断基準:
- 初期表示に不要なコンポーネント（モーダル・タブ切り替え・折り畳みコンテンツ等）
- 大きなサードパーティライブラリに依存しているコンポーネント
- "use client" が付いていて、特定操作後にのみ表示されるコンポーネント

出力形式:
- コンポーネントパス: 現在のインポート方法 → 推奨インポート方法
- 期待される削減効果（概算）
```

#### 4-2. 未使用 import の検出

```bash
cd webapp && npx knip --reporter compact 2>/dev/null || true
```

knip が使用できない場合は、以下の観点でサブエージェントに手動確認させる:
- import しているが実際には使用していない変数・型・コンポーネント
- re-export だけして使われていないファイル

#### 4-3. 画像の next/image + BASE_PATH 対応状況

Agent ツールで以下を実行する（`context: fork`）:

```
webapp/src/ 配下の全ファイルで画像の扱いを確認してください。

確認項目:
1. <img> タグを直接使用している箇所（next/image に置き換え対象）
2. next/image を使用しているが src に process.env.BASE_PATH が付与されていない箇所
3. CSS/style 属性で背景画像を指定しているが BASE_PATH が考慮されていない箇所

参照: CLAUDE.md（「画像パスは必ず process.env.BASE_PATH を先頭に付与すること」）

出力形式:
- ファイルパス:行番号 | 問題の種別 | 修正方法
```

#### 4-4. "use client" 境界の見直し提案

Agent ツールで以下を実行する（`context: fork`）:

```
webapp/src/ 配下で "use client" ディレクティブの境界が最適化されているか確認してください。

確認項目:
1. ファイル全体に "use client" が付いているが、インタラクティブな部分は一部のみのケース
   → クライアント部分を小さなコンポーネントに切り出せる
2. "use client" コンポーネントが大きな Server Component ツリーのルートになっているケース
   → クライアントの範囲を末端に絞れる
3. useState / useEffect を使っていない "use client" コンポーネント
   → "use client" を削除してサーバー側で実行できる

参照: CLAUDE.md・docs/dev/component-design.md

出力形式:
- ファイルパス | 現状 | 提案 | 期待効果
```

### 5. 監査レポートの出力

以上の分析結果をまとめて、以下の形式でレポートを出力する:

```markdown
## パフォーマンス監査レポート

### バンドルサイズ概要
- 合計サイズ: X KB
- JS 合計: X KB
- CSS 合計: X KB
- 要調査バンドル（>50KB）: N 件

### 重要度別 最適化提案

#### 高優先度
- ...

#### 中優先度
- ...

#### 低優先度（参考）
- ...

### 次のアクション
1. ...
2. ...
```
