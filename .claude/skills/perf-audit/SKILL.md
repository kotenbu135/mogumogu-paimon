# パフォーマンス監査スキル

静的エクスポート（`out/`）のバンドルサイズを分析し、優先度付きの最適化提案を出力する。

## 手順

### 0. 事前確認

```bash
ls webapp/out/_next/static/ 2>/dev/null && echo "EXISTS" || echo "MISSING"
```

`MISSING` の場合はステップ1でビルドを実行する。`EXISTS` でも内容が古い可能性があるため、ユーザーに再ビルドが必要か確認する（明示的に「最新状態で監査して」と言われた場合は自動でビルドする）。

### 1. ビルド実行（必要な場合）

```bash
cd webapp && npm run build 2>&1 | tail -30
```

ビルドエラーがある場合は監査を中断してエラーを報告する。Next.js がビルド時に出力するページ別サイズ情報（First Load JS）も合わせてキャプチャする。

### 2. バンドルサイズ集計

Agent ツールを `context: fork` で起動し、以下の分析を委譲する:

```
webapp/out/ ディレクトリ内の全 JS・CSS ファイルのサイズを集計してください。

手順:
1. out/_next/static/ 配下の全 .js・.css ファイルを列挙する（Glob）
2. 各ファイルのサイズ（バイト）を確認する（Bash: stat -c%s <file> または ls -la）
3. KB に変換してサイズ降順でソートしたリストを作成する
4. JS 合計・CSS 合計・全体合計を算出する
5. 30KB を超える JS ファイルを「要調査バンドル」として別途リストアップする

出力:
- ファイルパス | サイズ(KB) | 種別(JS/CSS) の一覧（降順）
- JS 合計 / CSS 合計 / 全体合計
- 要調査バンドル（>30KB）の一覧
```

### 3. 大きなバンドルの原因特定

ステップ 2 の結果を受け取り、要調査バンドルが存在する場合のみ実行する。Agent ツールで以下を実行する（`context: fork`）:

```
以下の要調査バンドルについて、原因コンポーネントを特定してください。

要調査バンドル:
<ステップ2の結果から要調査バンドルのファイルパスとサイズを列挙してここに記載>

手順:
1. webapp/src/ 配下の全 .tsx/.ts ファイルを調査する（Glob + Read）
2. 以下の観点で原因を特定する:
   a. "use client" ディレクティブが付いているファイルの一覧と依存パッケージ
   b. useState / useEffect を使っていないのに "use client" のファイル（Server Component 化できる）
   c. 大きなサードパーティライブラリ（lodash・date-fns・chart.js 等）をクライアント側で import しているファイル
   d. 動的インポート（next/dynamic）を使わず静的 import しているコンポーネントのうち、遅延ロードできるもの

出力:
- ファイルパス | 問題の種別 | 推定影響 | 対処法
```

### 4. 最適化チェック（並列実行）

以下の 3 つを Agent ツールで並列に実行する（それぞれ `context: fork`）:

#### 4-A. 動的インポート候補の特定

```
webapp/src/ 配下で next/dynamic による遅延ロードに適したコンポーネントを特定してください。

判断基準（いずれかに該当するもの）:
- 初期表示に不要（モーダル・ドロワー・タブパネル・アコーディオン等）
- 大きなライブラリに依存しており、特定操作後にのみ使われる
- "use client" かつ特定イベント後にのみ描画される

出力（該当がない場合はその旨を明記）:
- ファイルパス: 現在の import 文 → 推奨 next/dynamic 実装例
- 期待削減量（概算）

next/dynamic の実装例:
const HeavyComponent = dynamic(() => import('./HeavyComponent'), { ssr: false })
```

#### 4-B. 未使用 import の検出

```bash
cd webapp && npx --yes knip --reporter compact 2>/dev/null | head -50 || echo "KNIP_UNAVAILABLE"
```

`KNIP_UNAVAILABLE` の場合は以下をサブエージェントで実行する（`context: fork`）:

```
webapp/src/ 配下の全ファイルで未使用の import を確認してください。

確認項目:
- import しているが使用していない変数・型・コンポーネント
- 他のファイルから import されていない export（barrel ファイルを除く）

出力: ファイルパス:行番号 | 未使用の識別子 | 削除可否
```

#### 4-C. 画像・BASE_PATH の対応確認

```
webapp/src/ 配下で画像の扱いを確認してください。

確認項目（問題があるもののみ報告）:
1. <img> タグを直接使用している箇所（next/image 推奨）
2. next/image を使用しているが src に process.env.BASE_PATH が付与されていない
3. CSS / style 属性で backgroundImage を指定しているが BASE_PATH が考慮されていない

根拠: CLAUDE.md「画像パスは必ず process.env.BASE_PATH を先頭に付与すること」

出力（問題がない場合はその旨を明記）:
- ファイルパス:行番号 | 問題の種別 | 修正方法
```

### 5. 監査レポートの出力

以上の結果をまとめて以下の形式でレポートを出力する。**提案は必ず「すぐできる対応」から順に並べること**:

```markdown
## パフォーマンス監査レポート

### バンドルサイズ概要
| 種別 | サイズ |
|------|--------|
| JS 合計 | X KB |
| CSS 合計 | X KB |
| 全体合計 | X KB |
| 要調査バンドル（>30KB） | N 件 |

### 最適化提案

#### 高優先度（効果大・工数小）
- ...（具体的なファイルパスと修正内容を記載）

#### 中優先度
- ...

#### 低優先度（参考）
- ...

### 次のアクション（優先順）
1. `ファイルパス` の `use client` を削除して Server Component 化する
2. `ファイルパス` を next/dynamic で遅延ロードする
3. ...
```

提案には必ずファイルパスと具体的な変更内容を含めること。「検討してください」で終わらず、「このファイルのこの行をこう変える」まで踏み込む。
